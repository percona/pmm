PMM 3.7.0 introduces Real-time Analytics (RTA) API endpoints, for programmatic access to live query execution data on your MongoDB services.

## New features

### Real-time Analytics API (Technical Preview - MongoDB only)

While Query Analytics (QAN) helps you optimize performance by analyzing completed queries, the new RTA API shows you what's happening on your databases right now. 

When your database is struggling, you can use these endpoints to identify problematic operations immediately and take action before they impact users.

The RTA API provides two endpoints that let you monitor live query execution and control real-time monitoring for your MongoDB services and clusters.

#### New endpoints

- **GET /v1/realtime/query-data** — Returns currently executing queries
- **POST /v1/realtime/change** — Enables or disables RTA for services or clusters

#### GET /v1/realtime/query-data

Retrieves live query execution data from MongoDB services.

**Query parameters:**

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `cluster` | string | No | Filter results to a specific MongoDB cluster |
| `service` | string | No | Filter results to a specific MongoDB service |

#### Example request
```sh
curl -X GET "https://your-pmm-server/v1/realtime/query-data?service=mongodb-production-rs0" \
  -H "Authorization: Bearer glsa_xxxxx" \
  -H "Content-Type: application/json"
```

#### Example response
```json
{
  "queries": [
    {
      "service_id": "7a3e9c44-12ab-4d3f-9e21-5c8d7b1a2e4f",
      "service_name": "mongodb-production-rs0",
      "cluster": "production-cluster",
      "database": "orders",
      "collection": "transactions",
      "operation": "find",
      "query": "{\"status\": \"pending\"}",
      "duration_ms": 2547,
      "namespace": "orders.transactions",
      "client": "10.0.1.45:52341",
      "app_name": "order-processor",
      "plan_summary": "IXSCAN { status: 1, created_at: 1 }",
      "num_yields": 12
    }
  ],
  "timestamp": "2024-03-06T15:23:45Z",
  "total_queries": 1
}
```

#### POST /v1/realtime/change

Enables or disables Real-time Analytics for a MongoDB service or cluster.

**Request body (service-level):**
```json
{
  "service_id": "7a3e9c44-12ab-4d3f-9e21-5c8d7b1a2e4f",
  "enabled": true
}
```

**Request body (cluster-level):**
```json
{
  "cluster": "production-cluster",
  "enabled": true
}
```

**Note:** Exactly one of `service_id` or `cluster` must be specified.

#### Example request
```sh
curl -X POST "https://your-pmm-server/v1/realtime/change" \
  -H "Authorization: Bearer glsa_xxxxx" \
  -H "Content-Type: application/json" \
  -d '{
    "service_id": "7a3e9c44-12ab-4d3f-9e21-5c8d7b1a2e4f",
    "enabled": true
  }'
```

#### Example response
```json
{}
```

### Key capabilities

- **Live query monitoring**: See currently executing queries as they happen
- **Flexible filtering**: Filter by service or cluster for targeted monitoring
- **Granular control**: Enable/disable RTA per service or across entire clusters
- **Rich query details**: Access operation type, duration, plan summary, lock information, and client details

### Use cases

- **Incident response**: Identify problematic queries during active performance issues
- **Cluster-wide visibility**: Monitor all services in a MongoDB cluster simultaneously
- **Automated monitoring**: Integrate real-time query data into custom dashboards and alerting systems
- **Troubleshooting**: Correlate slow operations with client applications and database activity

### Requirements

- PMM Server 3.7.0 or later
- PMM Client 3.7.0 or later
- MongoDB service monitored by PMM
- Configured MongoDB exporter

### Limitations

- **Technical Preview**: Not recommended for production use
- **MongoDB only**: MySQL and PostgreSQL support planned for future releases
- **Ephemeral data**: Query data is not persisted; it reflects current execution only
- **Performance impact**: Real-time monitoring adds overhead to MongoDB and PMM

### Impact

These are new endpoints with no impact on existing PMM installations or API integrations. RTA must be explicitly enabled for each service or cluster using the `/v1/realtime/change` endpoint.
